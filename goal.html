Build Agent Breach Replay

This page is the working goal for the whole project. It gives a coding agent enough context to build the product end to end: SDK, ingestion, replay studio, privacy model, detections, OpenAI explanations, Moss retrieval, and deployment path.

Mission

What We Are Building

A privacy-first security trace layer and visual replay studio for tool-using AI agents. It captures security-relevant events and replays how untrusted content influenced privileged tool actions.

What It Should Prove

Teams should be able to see where untrusted content entered, how it flowed through the agent, which tool boundary was crossed, and what policy or guardrail would have stopped the unsafe chain.

Product Flow

Trace SDK

Captures source, tool, policy, and violation events.

Ingestion

Validates schema and stores metadata-only traces.

Detectors

Runs deterministic security predicates.

AI + Moss

Explains findings and retrieves similar incidents.

Replay Studio

Visualizes the breach path and exports reports.

Agent Prompt

You are building Agent Breach Replay, a privacy-first security trace layer and visual replay studio for tool-using AI agents.

Goal:
Build a real release-quality product that helps teams debug AI-agent security failures. The product must show how a trusted user task, untrusted content, agent planning, tool calls, protected data, policy decisions, and external actions combine into a replayable security incident.

Core product:
1. Agent Breach Trace SDK
   - TypeScript SDK first, Python SDK second.
   - Capture security events: trace.start, source.read, model.step, tool.call, policy.decision, violation.detected, trace.end.
   - Track trust, authority, data class, source influence, tool boundary, destination class, and policy decision.
   - Default to metadata-only capture.
   - Redact locally before upload.
   - Hash sensitive content instead of storing raw values.

2. Replay Studio
   - Build with Next.js, React, TypeScript, Tailwind, shadcn/ui, React Flow, and Framer Motion.
   - Provide a dashboard of agent runs.
   - Provide a step-by-step replay timeline.
   - Provide a source-to-tool influence graph.
   - Provide a security predicate panel.
   - Provide a trace inspector.
   - Provide a report export view.
   - Provide similar incident search powered by Moss.

3. Backend and data:
   - Use Postgres as the source of truth for orgs, projects, agents, runs, trace events, findings, policies, and reports.
   - Use Moss as a complementary semantic retrieval layer for incident summaries, trace narratives, guardrail docs, and similar incident search.
   - Use OpenAI models for incident explanations, guardrail recommendations, report generation, and optional trace normalization assistance.
   - Do not use the LLM as the only detector for serious security findings.
   - Deterministic rules must trigger findings first. OpenAI models explain findings second.

4. Initial detections:
   - Exfiltration: protected data reaches an external destination.
   - Untrusted-to-action: untrusted content influences a privileged or real-world action.
   - Confused deputy: an attacker causes the agent to misuse its authority.
   - Destructive write: the agent mutates or deletes data through an unsafe influence chain.

5. Integrations:
   - Build an OpenAI Agents SDK adapter.
   - Support JSON trace import/export.
   - Later support LangGraph, LangChain, LangSmith, and OpenTelemetry-style traces.
   - Do not compete with OpenAI tracing. Complement it with security semantics and replay.

6. Privacy requirements:
   - The default capture mode is metadata-only.
   - Raw emails, files, prompts, secrets, and customer records must not be required.
   - Support redacted previews only when explicitly enabled.
   - Support self-hosted or private-cloud deployment paths later.
   - Product promise: show the breach path without collecting the breached data.

7. Deployment:
   - Deploy the web app on Vercel.
   - Use managed Postgres such as Neon or Supabase.
   - Publish TypeScript SDK to npm.
   - Publish Python SDK to PyPI.
   - Host docs with the app or a docs framework later.

Implementation priority:
1. Create the trace schema and sample traces.
2. Build the TypeScript SDK mock that emits replay-compatible traces.
3. Build deterministic detectors.
4. Build the replay studio UI.
5. Add OpenAI-generated incident reports.
6. Add Moss-powered similar incident retrieval.
7. Add persistence with Postgres.
8. Add OpenAI adapter.
9. Add Python SDK.

Quality bar:
- Keep the product focused on security replay, not generic observability.
- Make privacy visible in the UI.
- Keep the UI dense, professional, and useful for engineers and security teams.
- Avoid overclaiming. Do not claim to prevent all attacks or fully secure all agents.
- Make every finding explainable through trace events, source labels, tool boundaries, and policy rules.
- The main demo should clearly show: untrusted source -> agent influence -> privileged tool -> security boundary -> violation -> recommended guardrail.

Working Principle

Agent Breach Replay does not infer secret model reasoning. It reconstructs security-relevant influence from explicit trace events, tool inputs, source labels, data classifications, and policy boundaries.
